# 信通畅邮台账/日志定制开发方案

> 基于《柒·台账、日志管理法则》合规要求 + 3 项新需求，对信通畅邮平台二次开发。
> **状态：方案待审批** | 编写：2026-06-30

## 一、背景

### 合规驱动
短信业务监管要求建立标准化**台账 + 日志**管理体系：全流程可追溯、不可篡改、留存≥6个月、可审计、权限管控。

### 平台现状
信通畅邮已部署（admin 8889 / enterprise 8890 / netway 8888，Java 8 + Tomcat 9.0.119 + MariaDB 10.11，构建/部署脚本就绪）。经评估：**日志能力较强(88%)，台账能力有结构性缺口**。本方案补齐缺口 + 落实 3 项新需求。

## 二、需求清单

|------|------|------|------|

## 三、现状覆盖度评估

| 模块 | 覆盖 | 平台对应 | 主要缺口 |
|------|------|---------|---------|
| 日志 8 项 | 🟢 88% | input_log/inbox/report/system_log | 第8项拒收无专门记录 |
| 短信业务台账 | 🟡 75% | input_log/report/sms_template | 投诉处理无表、签名审核弱 |
| 合作方台账 | 🟡 50% | enterprise 表 | 信用代码/资质有效期/经营范围/期限 |

合规机制：按日分表支持长期留存 ✅ / 检索索引 ✅ / 导出 ✅ / 权限体系 ✅ / 内容明文未加密 ⚠️ / 无自动清理 ⚠️ / 无防篡改 ⚠️

### 4.4 通用机制

- **时限清理定时任务**：扫描 code 表配置，按月 DROP sms_send 超期分表 + 清理超期台账/凭证（R3 落地，R4/R5 共享）
- **到期预警**：资质有效期/合作期限/码号许可临近到期 → 告警（复用 alarm 机制）
- **不可篡改**：日志表(input_log/report/inbox/system_log) DB 权限收紧为 INSERT+SELECT；查询/导出操作写 system_log 留痕
- **导出/打印归档**：各台账列表加导出（复用 export_file）
- **权限管控**：台账/日志菜单靠 admin_limit 角色权限控制访问

## 六、待确认项（需你拍板）

1. **操作员权限边界**（R2）：操作员能做什么、不能做什么？（建议：能发短信+查记录，不能改通道/审核/看财务）
2. **签名审核流程**（R1）：签名谁审核？未审核签名是否禁止发送？
3. **各时限具体值**（R3）：日志/台账/同意凭证各留几个月？销毁是物理删除还是先归档？
4. **字段加密**（R4-D）：监管是否强制日志内容字段加密？（影响检索，工作量大）
5. **开发方式**：直接改服务器源码 `/tools/sms/sms_platform`，还是本地开发后部署？是否引入 git 版本管理？
6. **测试数据**：当前 DB 有演示数据（test 角色等），开发前是否清理？

## 七、开发流程与风险

### 流程
1. 确认待确认项（第六节）
2. 按批次开发：建表→Controller/Service/JSP→菜单数据→构建(deploy-sms.sh)→重启→验证
3. 每批次独立验证（登录平台测功能 + 看日志无报错）

### 风险
- **改现有表**(enterprise/product)：加字段安全(ALTER ADD)，但现有页面要同步改录入/展示
- **加密影响检索**(R4-D)：LIKE 搜索失效，需权衡
- **定时任务误删**：时限清理任务需充分测试，避免误删未到期数据
- **二次开发质量**：信通畅邮是开源学习项目，代码质量一般，定制需适配其 Spring MVC + JSP + MyBatis + 自定义标签体系

### 建议先做的最小验证
第1批（操作员角色）**零代码、零风险**，先做它验证权限体系跑通，再推进后续有代码的批次。
