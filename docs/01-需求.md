# 需求（采访结论）

> 2026-06 通过逐题采访确认。新会话不要重新询问这些，如需扩展再问。

## 业务定位

**短信中间商平台**。用户本人是中间商：
- 对下：甲方（企业客户）通过 HTTP 协议提交短信
- 对上：连接运营商 + 上游 MSP 实际发送
- 中间：路由、计费、客户管理、状态回推

## 需求逐条（采访 Q1–Q6）

| 问题 | 选择 | 含义 |
|------|------|------|
| Q1 用途 | **A + C** | 短信业务运营（卖短信/计费/多级代理）+ 网关聚合（SMPP 等协议层） |
| Q2 市场 | **A 国内为主** | 必须支持 CMPP/SGIP/SMGP（国内三件套） |
| Q3 技术栈 | 语言不限 | 能部署能用即可 |
| Q4 通道来源 | **C 混合** | 部分直连运营商（CMPP/SGIP/SMGP）+ 部分对接上游 MSP（HTTP） |
| Q5 规模 | **C 大规模** | 日发千万级，需高并发协议网关 + 分布式架构 |
| Q6 开源态度 | **A 必须纯开源** | 接受风险，不接受半商用/源码授权 |

## 核心约束

1. **国内协议必备**：CMPP（移动）/ SGIP（联通）/ SMGP（电信）+ SMPP
2. **混合通道**：同时支持直连运营商 + HTTP 上游 MSP
3. **完整运营功能**：客户管理、计费充值、多级代理、路由分拣、黑白名单、敏感词、状态报告
4. **大规模**：千万级日发，高并发，可扩展
5. **纯开源**：不接受商业授权

## 验收方向（功能层面）

- [ ] 甲方能通过 HTTP API 提交短信
- [ ] 甲方有 Web 客户端登录发短信
- [ ] 平台能连运营商 CMPP 通道下发
- [ ] 平台能连 HTTP 上游 MSP 下发
- [ ] 路由分拣（按号段/运营商分流到不同通道）
- [ ] 计费、充值、余额
- [ ] 黑名单、敏感词过滤
- [ ] 状态报告回推给甲方
- [ ] 发送记录查询、报表

> 注：以上是方向，未细化正式验收标准。部署跑起来后再逐项验证。
