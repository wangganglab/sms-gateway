# 选型决策（ADR）

## 背景

需求：国内短信中间商平台（对下甲方 HTTP，对上运营商 CMPP/HTTP + HTTP 上游 MSP），大规模，纯开源。在 GitHub + Gitee + 搜索引擎上做了两轮搜索。

|------|------:|------|------|------------------|---------------------|----------|
| **信通畅邮** liuyanning/sms_platform | 33 | 一般 | ❌ SQL不兼容(可修) | ✅ 全(CMPP/SGIP/SMGP/SMPP) | ✅ 完整 | ✅ **中间商平台** |

## 决策：选信通畅邮（liuyanning/sms_platform）

## 原因

1. **角色完全匹配**：它是中间商平台——sms-enterprise（甲方客户端）+ sms-admin（运营管理）+ sms-netway-sender（网关），正是"对下甲方 + 对上运营商"的结构
2. **国内协议齐全**：CMPP2.0/3.0、SGIP、SMGP、SMPP 全支持（上下游都支持）
3. **混合通道支持**：同时支持直连运营商（CMPP）+ HTTP 上游（自研HTTP/迈远http），匹配 C 方案
4. **运营功能完整**：客户/通道/计费/充值/黑白名单/敏感词/路由/报表/彩信

## 取舍（诚实说明）

- **star 数低（33）**：社区小，文档主要靠 README + QQ群(533147696)
- **SQL 兼容问题**：business.sql 在 MariaDB 10.11 报 Row size too large —— 已定方案（sed 改 ROW_FORMAT），非硬伤
- **单机版**：README 明确"仅供学习，禁止商用，单机版有性能局限"。大规模（千万级）需评估，可能要二次开发分布式
- **协议中间件依赖**：依赖自研 quick-sms（gitee.com/zhuang-dian/quick-sms），滑动窗口控速

## 排除理由

- **playSMS**：质量好能跑通，但不支持国内协议，国内中间商用不了 → 已弃用
- **austin**：质量最高最活跃，但角色是"发送方/客户"（向上游服务商发短信），不提供甲方 HTTP 入口、不连运营商，与中间商定位**完全相反**
- **qfjiaoyan/sms-platform**：协议全 + 分布式，但无 SQL 脚本，部署门槛更高；先信通畅邮跑通后再考虑
- **simple-sms / Jasmin**：只是协议网关/中间件，无平台层，需大量自研

## 后果

- 短期：信通畅邮能快速跑起来一个功能完整的中间商平台
- 中期：大规模场景需评估单机瓶颈，可能基于 quick-sms 做分布式扩展
- 长期：纯开源 + 低 star = 要做好自己填坑/二次开发的准备
